# How We Built the VA AA Districts Map

## The Problem

Virginia Area 71 AA has 48 districts, but their boundaries don't follow county lines — especially in dense areas like Northern Virginia, where a single county (Fairfax) contains meetings from 9 different districts. The original plugin (generated by Perplexity Computer) assigned each Virginia county to exactly one district, which was a rough approximation at best.

## Data Source: The Meetings API

The aavirginia.org site runs the [12 Step Meeting List](https://wordpress.org/plugins/12-step-meeting-list/) WordPress plugin, which exposes a public JSON API:

```sh
curl "https://aavirginia.org/wp-admin/admin-ajax.php?action=meetings"
```

This returns ~2,650 individual AA meeting records, each with:
- `latitude`, `longitude` — exact GPS coordinates
- `district` — tagged string like `"District 21 - Springfield"`
- `formatted_address` — full street address
- `region` — city/area name

This is the ground truth for where each district's meetings actually are.

## Step 1: Reverse-Geocode Meetings to Counties

We loaded Virginia's county/city boundaries from the same TopoJSON the map uses (`us-atlas@3/counties-10m.json`), decoded the arcs into Shapely polygons, and ran point-in-polygon for each meeting's lat/lng to determine which county it falls in.

This produced a mapping of `county FIPS -> {district: meeting_count}`. For example, Fairfax County (FIPS 51059) had:
- District 21: 87 meetings
- District 6: 86 meetings
- District 4: 61 meetings
- District 11: 60 meetings
- District 42: 28 meetings
- ...and 4 more districts

We assigned each county's primary district as the one with the most meetings, and flagged counties with significant presence from multiple districts as "shared."

**Result:** 32 county reassignments from the original mapping, and 12 properly identified shared counties.

## Step 2: Filter Out Mistagged Meetings

Some meetings in the API have incorrect district tags. For example, "Happy Destiny" is tagged as District 24 (Western Tidewater) but is actually located in Charlottesville — squarely inside District 9 territory.

We detected these by computing each district's centroid (average lat/lng of all its meetings), then flagging meetings that were:
1. More than 3x closer to another district's centroid than their own
2. AND more than ~55km from their own centroid

This caught 8 mistagged meetings (including 5 copies of "Just For Today" tagged Norfolk/District 14 but located in Charlottesville, and one Winchester meeting tagged in Virginia Beach). Without this filtering, these outliers would create spurious district "islands" in the Voronoi diagram.

## Step 3: Exclude Non-Geographic Districts

District 48 (Spanish Speaking) is statewide — it has meetings scattered across the state but doesn't correspond to a geographic area. Its 3 meetings in Amelia Court House would have created a small polygon in the middle of the map. We excluded it from boundary generation; its area was absorbed by neighboring District 29 (South of the James).

## Step 4: Generate Voronoi District Boundaries

With clean, geocoded meeting points grouped by district, we generated sub-county boundaries using [Voronoi diagrams](https://en.wikipedia.org/wiki/Voronoi_diagram):

1. **Deduplicate points** — Collapse meetings at nearly-identical coordinates (within ~10m) to avoid Voronoi degeneracies. This reduced ~2,030 meetings to 708 unique points.

2. **Compute Voronoi** — Using Shapely's `voronoi_diagram()`, we created a cell around each meeting point. Each cell contains all geographic area closer to that meeting than to any other meeting.

3. **Assign cells to districts** — Each Voronoi cell inherits the district of the meeting point it contains.

4. **Merge by district** — All cells belonging to the same district are merged into a single (possibly multi-part) polygon using `unary_union()`.

5. **Clip to Virginia** — The merged polygons are intersected with Virginia's state boundary to trim off area outside the state.

**Result:** 41 district boundary polygons stored as GeoJSON in `district-boundaries.json` (65KB).

### Why Voronoi Works Here

The Voronoi approach is a natural fit because it answers the question: "Given a point on the map, which district's meetings are closest?" This produces boundaries that:
- Follow the actual geographic spread of meetings, not arbitrary county lines
- Automatically handle dense areas like NoVA where 12 districts share one county
- Give larger areas to districts with spread-out rural meetings
- Create contiguous boundaries (no gaps, no overlaps) that tile the entire state

### Limitations

- Districts with few meetings (e.g., District 17 has only 1 meeting) get boundaries determined entirely by their neighbors' meeting locations
- The boundaries are only as accurate as the meeting locations — if meetings are added or moved, the boundaries should be regenerated
- Some districts' true boundaries may follow roads or landmarks that the Voronoi can't capture

## Step 5: Render as Map Overlay

The plugin renders the boundaries as a semi-transparent colored overlay on top of a neutral county base:

- **Base layer:** Virginia counties filled with a flat neutral color, no strokes (just provides the shape)
- **Boundary layer:** Voronoi district polygons filled with pastel colors, white strokes between them
- **Labels:** District numbers with white halo (`paint-order: stroke`), counter-scaled on zoom to stay constant screen size, with collision detection that hides overlapping labels and reveals them as you zoom in

The county base layer is kept for click/hover detection (each county knows its primary district via the FIPS mapping), while the Voronoi overlay provides the visual boundaries.

## Tools Used

- **Python** with Shapely, scipy — geometry processing, Voronoi generation, point-in-polygon
- **D3.js v7** — SVG map rendering, zoom, interaction
- **TopoJSON** — Virginia county/state boundary data (us-atlas@3)
- **WordPress** — plugin framework, data storage (wp_options), admin UI
- **Docker** — local WordPress development environment
